# ZRONG 优化迭代方案 v29

## 目标
- 提高输出节点中国大陆可用率
- 保证项目稳定性（Actions 60分钟内完成）
- 增强可维护性（模块化、配置外置）
- 每轮优化→验证→修复→再优化循环

## 问题诊断

### P0 - 严重（直接影响可用率）
1. **节点命名崩溃**: 所有150节点名称无亚洲标识，🇹🇷1-993mss0.0格式无法识别区域
2. **速度测量无效**: test_proxy返回speed=0.0或虚假204000.0，score参数传递错误
3. **亚洲配额未生效**: 0/150节点有亚洲标识，TARGET_ASIA_RATIO=0.45完全失效

### P1 - 重要（影响稳定性/可维护性）
4. **main_flow.py仍686行**: 核心逻辑混合采集、排序、输出，难以维护
5. **filter排除关键词过宽**: "免费/公益/临时"可能误杀有效的亚洲节点
6. **测速URL策略**: 只测204响应无法验证真实可用性，应增加内容验证
7. **TCP补充阈值宽松**: 非亚洲延迟800ms也能进入最终输出

### P2 - 改进（锦上添花）
8. **历史数据未利用**: 节点历史可用性数据采集但评分权重低
9. **源权重反馈闭环缺失**: 应根据最终输出中源的占比动态调整权重
10. **缺少自动可用率检测**: 每次运行后应自动检测输出节点的大陆可用率

## 迭代计划

### Phase 1: 修复命名器+测速 (v29.01) ← 当前
- 修复NodeNamer.generate()参数传递
- 修复score/speed传递链
- 修复节点名称格式：emoji+区域-编号-延迟ms-速度KB/s
- 增加真实内容验证URL（baidu.com首页）

### Phase 2: 优化采集+过滤 (v29.02)
- 缩小filter排除关键词范围
- 增加大陆友好协议优先策略（VLESS+Reality > VLESS+WS > Trojan > VMess）
- 优化亚洲节点识别逻辑

### Phase 3: 优化测速+排序 (v29.03)
- 增加真实下载测速（下载10KB文件测速而非204响应）
- 优化final_sort_key权重
- 亚洲节点延迟阈值收紧（TCP补充≤500ms而非800ms）
- 增加历史可用性权重

### Phase 4: 模块化重构 (v29.04)
- main_flow.py拆分：采集→验证→排序→输出 四个独立函数
- 配置外置到config/rules.yaml（所有阈值/权重/关键词）
- 增加自动可用率检测模块

### Phase 5: 稳定性+监控 (v29.05)
- Actions运行时间优化（减少无效测试批次）
- 增加运行后可用率自动检测
- 失败自动回滚机制
- 输出节点可用率≥30%目标

## 自动化执行
- 每轮修改→本地py_compile验证→git commit→push→触发Actions
- Cron监控Actions结果
- 循环直到可用率达标+运行稳定
