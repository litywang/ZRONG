## 一、问题诊断

### 1.1 已确认问题（按影响排序）

| # | 问题 | 根因 | 影响 | 状态 |
|---|---|---|---|---|
| P0-1 | TEST_URLS 含不可达HTTP URL | ipip.net/ip.3322 需重定向 | 节点误判失败 | ✅ v28.98已修复 |
| P0-2 | `allow_redirects=False` 阻止302重定向 | HTTP→HTTPS重定向被拦截 | 主池失效 | ✅ v28.98已修复 |
| P0-3 | 健康检查在Actions高并发下超时 | Clash API 150节点8并发不稳定 | Run #494失败 | ✅ v28.98已禁用 |
| P0-4 | Gist step `if: secrets.GIST_ID != ''` 条件不可靠 | secret存在时表达式行为不确定 | 整个step失败 | ✅ v28.98已修复 |
| P1-1 | 速度计算用总延迟替代纯下载时间 | 复用首次请求时间 | speed数值虚高 | ✅ v28.98已修复 |
| P1-2 | test_proxy 重试后不等待连接释放 | 立即重试可能用同一连接 | 重试无意义 | 🔲 待优化 |
| P1-3 | MAX_PROXY_LATENCY=5000ms 阈值宽松 | Actions环境延迟高 | 劣质节点混入 | 🔲 待观察 |
| P2-1 | 备用池 TEST_URLS_BACKUP 过多备用URL | 国内优先策略 | 备用池失效 | 🔲 待优化 |
| P2-2 | 未利用 Clash 的 Provider / Proxy Group | 直接替换 proxies | 无法按组筛选 | 📋 设计中 |
| P3-1 | main_flow.py ~700行，无单测 | 大函数+无测试 | 可维护性差 | 📋 计划中 |

### 1.2 核心优化思路

```
目标：输出节点大陆可用率最大化
约束：稳定性优先（Actions 60分钟超时）
方法：分层验证 + 宽松准入 + 严格排序
```

## 二、验证体系重构

### 2.1 当前验证链（v28.98）

```
采集 → TCP Ping → 协议握手 → Clash测速 → 排序输出
```

### 2.2 优化后验证链（目标）

```
采集（去重+黑名单）
  ↓
TCP Ping（快速过滤）
  ↓
协议握手（确认协议有效）
  ↓
Clash测速（真实代理性能）
  ↓
健康检查（Clash API，150节点时启用）
  ↓
排序输出（亚洲优先+速度+历史稳定性）
```

### Phase 2：可用率提升（v28.99~v29.10）
**目标：大陆可用率 >85%（目标90%）**

| 优化项 | 预期效果 | 风险 |
|---|---|---|
| TEST_URLS 增加 gstatic + cloudflare 为主 | 减少误判 | 低 |
| 启用 `ENABLE_MAINLAND_TEST`（可选） | 精确大陆可达性 | 中 |
| MAX_PROXY_LATENCY 收紧（5000→3000） | 过滤劣质节点 | 低 |
| 历史稳定性分数权重提升 | 保留稳定老节点 | 低 |
| **重试机制优化（退避+连接复用）** | 减少误判 | 中 |

### Phase 3：健康检查恢复（v29.xx）
**目标：150节点时安全启用Clash API健康检查**

触发条件：
1. 单次运行合格节点 >120（节点充足时）
2. 并发降为 4，timeout=15s
3. 每批次最多测10个节点

### Phase 4：可维护性提升（v30.xx）
- main_flow.py 拆分（采集/验证/评分/输出）
- 添加 pytest 单元测试
- 配置文件外置（rules.yaml / urls.yaml）
- 日志分级（INFO/WARN/DEBUG）
